# ReconfCNN
This project presents an FPGA-based user-configurable Convolutional Neural Network (CNN) accelerator implemented as a closed-loop system. The architecture is designed to efficiently execute convolution operations with flexible reconfigurability.

## Architecture Documentation

### Processing Element (PE)

The Processing Element (PE) is the fundamental computation unit of the accelerator. Each PE consists of:

- **One FP16 multiplier**: Performs element-wise multiplication between input feature values and kernel weights
- **Two FP16 adders**: Accumulate partial results

This combination enables each PE to efficiently compute a portion of the convolution operation in parallel with other PEs.

### Convolution Engine (CE)

The Convolution Engine is the core computational block responsible for performing convolution operations. It is composed of an **8 × 8 array of Processing Elements (PEs)**, allowing parallel computation across multiple data points.

To correctly perform convolution, the CE uses two key shifting mechanisms:

#### Horizontal Shift (hor_shift)

The horizontal shift ensures that values within the same row are properly aligned and accumulated. As input data moves horizontally across the PE array, each PE contributes to the accumulation of results corresponding to a sliding window over the input feature map. This shifting enables efficient reuse of data within rows and supports continuous streaming of inputs without recomputation.

#### Vertical Shift (ver_shift)

The vertical shift ensures that values aligned across rows (i.e., elements belonging to the same kernel window vertically) are accumulated correctly. By shifting data vertically across the PE array, the engine combines contributions from different rows of the kernel.

Together, horizontal and vertical shifts enable the CE to replicate the sliding window behavior of convolution, ensuring that all relevant input and kernel values are multiplied and accumulated correctly.

### Super Convolution Engine (Super CE)

The Super Convolution Engine extends the functionality of the standard Convolution Engine. While it retains the same architecture and capability to perform convolution, it also supports additional operations such as:

- **Activation functions** (e.g., ReLU)
- **Pooling operations** (e.g., max pooling)

This makes the Super CE more flexible, allowing multiple stages of a CNN pipeline to be executed within the same hardware block, reducing data movement and improving overall efficiency.

### Wrapper for CE and Super CE (Engine Shift Mechanism)

The wrapper module connects multiple Convolution Engines (or Super CEs) and introduces an additional shifting mechanism known as the **engine shift**.

Since convolution often involves multiple input channels (depth), partial results must be accumulated across these channels. The engine shift works as follows:

1. One CE processes a portion of the input depth and produces partial results
2. These results are passed to the next CE
3. The next CE accumulates its own results with the received partial sums
4. This process continues across multiple engines until the full depth is processed

This mechanism enables efficient accumulation across channels without requiring excessive memory access, forming a scalable and modular design.

### Mask Generator

The Mask Generator identifies which outputs from the 8 × 8 PE array are valid.

Due to varying kernel sizes and stride values, not all computed outputs correspond to valid convolution results. The Mask Generator generates a mask that flags valid positions based on the current configuration, ensuring only meaningful outputs are considered.

### Valid Data Module

The Valid Data module applies the mask generated by the Mask Generator.

- **Valid outputs** are retained
- **Invalid outputs** are replaced with zeros

This ensures that only correct convolution results are propagated forward, simplifying further processing stages.

### Memory Architecture

The memory system is designed to efficiently feed data into the Convolution Engines.

The system contains:

- **Three buffers**, with each buffer serving one Convolution Engine
- **15 memory banks** per buffer
- **32-bit width** per bank, allowing it to supply data to four PEs simultaneously (since each PE requires 16-bit FP16 inputs)

The depth of each memory bank is configurable and depends on available FPGA resources and application requirements. This banked memory structure enables high bandwidth and parallel data access, which is critical for maintaining throughput in the accelerator.

### Controller (FSM-Based)

The Controller is the central control unit of the system and is implemented as a Finite State Machine (FSM).

Its responsibilities include:

- **Managing** the flow of input feature maps and weights
- **Generating** control signals for all shifting operations (horizontal, vertical, and engine shift)
- **Coordinating** memory reads and writes
- **Synchronizing** computation across all modules

The controller is reconfigurable, meaning it can adapt to different convolution parameters such as kernel size, stride, and input dimensions. This flexibility allows the accelerator to support a wide range of CNN architectures without requiring hardware redesign.

### Weight Memory

The Weight Memory module is responsible for storing and distributing convolution kernel weights.

- **Broadcasts weights** efficiently to all relevant PEs
- **Aligned** with the required output channel depth, allowing it to automatically provide the correct weights for each channel
- **Eliminates** the need to manually address weights for every kernel and output channel, simplifying control logic and improving performance.
